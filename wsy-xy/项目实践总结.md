# 项目实践总结

本项目目标是把“网商园选品/列表页 → 单品图片下载 → AI 生成文案 → 保存 info.json → 闲鱼自动发布”串成一条可批量执行的自动化流水线，并提供一个尽量简单的前端界面用于投喂 URL 和观察日志。

## 1. AI Agent 设计思想

### 1.1 分层与边界

- **业务编排层（Agent/编排器）**：负责“做决策”，即决定下一步该调用哪个模块、何时重试、何时降级、如何处理异常与继续/跳过。
- **工具执行层（Tools/模块）**：负责“干活”，即对外部世界产生副作用的动作：浏览器自动化、文件读写、网络请求、图片下载、发布点击等。

这样拆分的好处是：

- 复杂性可控：工具保持单一职责，编排器集中处理流程控制与容错。
- 可替换：某个工具实现变化（比如页面 DOM 变化、接口变化）时，只需要替换工具，不破坏编排逻辑。
- 易观测：编排器统一产出日志/状态，前端只需要渲染日志即可。

### 1.2 状态与幂等

- **登录态持久化**：将登录态保存为 `wsy_state.json` / `xianyu_state.json`，后续运行尽量做到无需人工介入。
- **幂等策略**：同一商品重复执行时，目录结构固定（`products/商品ID_xxx/`），写入 `info.json` 覆盖即可；发布步骤在 UI 重载/刷新后具备“补传/自愈”的能力，降低失败概率。

### 1.3 面向失败的设计（Retry / Fallback / Continue）

- **外部 API 不稳定**（如 DeepSeek 超时）：采用重试 + 指数退避；最终失败时使用兜底文案，避免把错误字符串写入描述字段，确保流水线不停。
- **页面刷新/重渲染**：对 Playwright 的 page 引用做自愈，必要时重新获取活动页并补做关键步骤（例如图片上传确认）。
- **批量任务容错**：单个商品失败不阻塞后续商品，记录错误并继续处理下一个。

## 2. 工具调用流程

以批量发布为例，完整的“工具链”是：

1. **crawl**：在列表页提取商品详情 URL（按页面出现顺序），必要时等待登录完成并保存登录态。
2. **download_images**：打开商品页，抓取图片地址并下载到 `products/商品ID_xxx/`。
3. **generate_description**：调用 DeepSeek 生成描述（重试+降级）。
4. **save_product_info**：保存 `info.json`（便于回溯与后续复用）。
5. **upload_to_xianyu**：打开闲鱼发布页，上传图片、填写标题/描述/价格并发布（含刷新自愈与图片补传）。

运行时序（串行批处理）：

- 编排器先拿到商品 URL 列表
- 对每个商品按上述 1→5 顺序执行
- 每个商品执行结束（成功或失败）后进入下一个商品
- 全程用统一 logger 输出日志，前端只负责展示

## 3. 如何体现“AI 做决策，工具干活”的原则

### 3.1 AI（编排器）负责的决策

- 根据输入（列表页 URL、限制数量、价格策略）决定处理集合与执行顺序。
- 根据运行时信号（超时、页面缺元素、接口失败）决定：
  - 是否重试（次数、退避时间）
  - 是否降级（使用兜底文案/跳过某商品）
  - 是否继续下一个商品
- 统一汇总日志与结果状态（done/error），供 UI 展示与追踪。

### 3.2 工具（模块）负责的执行

- Playwright 打开页面、等待元素、抓取 DOM、点击按钮、上传文件。
- requests 下载图片、调用外部 API。
- 文件系统读写：创建目录、保存 `info.json`、保存/加载登录态文件。

### 3.3 “控制面”与“数据面”分离

- **控制面**：流程控制、状态机、重试/降级策略、批量调度（AI 逻辑）。
- **数据面**：网络 IO、浏览器 IO、文件 IO（工具执行）。

这种分离保证了“AI 不直接做 IO；工具不负责业务决策”，从而让系统更稳定、可维护、可扩展。
